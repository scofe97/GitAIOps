# 4.3 로그 수집 도구 선택

## 상황

Prometheus로 메트릭은 수집하고 있지만, "에러가 발생했다"는 것만 알 뿐 "어떤 에러인지"는 모른다. Pod 로그를 보려면 `kubectl logs`를 직접 실행해야 하고, Pod이 재시작되면 이전 로그가 사라진다. 중앙화된 로그 수집이 필요하다.

## 독자가 물어볼 수 있는 질문

- "로그를 한곳에서 보고 싶은데 뭘 써야 해?"
- "Pod이 재시작되면 로그가 날아가는데 어떻게 보관해?"
- "ELK Stack이랑 Loki 중에 뭐가 좋아?"

## 추천: Loki + Fluent Bit

Loki가 로그를 저장하고, Fluent Bit가 모든 노드에서 컨테이너 로그를 수집하여 Loki에 전송한다. Grafana에서 로그를 조회한다.

이 프로젝트에 적합한 이유:
- **경량**: Loki 128Mi, Fluent Bit 64Mi — e2-medium에서 ELK(2Gi+)는 불가능
- **Grafana 통합**: 메트릭(Prometheus)과 같은 UI에서 로그를 조회
- **라벨 기반 인덱싱**: 풀텍스트 인덱싱 대비 저장 비용이 낮다

## 비교

| 도구 | 특징 | 장점 | 단점 | Notiflex 적합도 |
|------|------|------|------|----------------|
| **Loki + Fluent Bit** | 라벨 인덱싱, Grafana 통합 | 경량(128Mi), 비용 낮음, Grafana 네이티브 | 풀텍스트 검색 느림 | ★★★ |
| **ELK Stack** | Elasticsearch + Logstash + Kibana | 풀텍스트 검색 강력, 성숙한 생태계 | 메모리 2Gi+, 운영 복잡 | ★ |
| **CloudWatch Logs** | AWS 네이티브 | 관리 불필요, AWS 통합 | GKE에서 사용 불편 | ★ |
| **Google Cloud Logging** | GCP 네이티브 | GKE 자동 수집, 무료 티어 | 커스텀 쿼리 제한, Grafana 미통합 | ★★ |

> **Notiflex 맥락**: e2-medium(4GB) 노드에 이미 Prometheus, Grafana, ArgoCD가 올라가 있다. Elasticsearch(최소 2GB)를 추가할 여유가 없다. Loki는 128Mi로 충분하고, 4.2에서 설치한 Grafana에 데이터소스만 추가하면 된다.

## 핵심 개념

- **DaemonSet**: Fluent Bit는 DaemonSet으로 배포된다. 모든 노드에 정확히 1개씩 Pod이 생성되어, 해당 노드의 모든 컨테이너 로그를 수집한다.
- **라벨 기반 인덱싱**: Loki는 로그 내용을 인덱싱하지 않고, `{namespace="notiflex", pod="api-xxx"}` 같은 라벨만 인덱싱한다. 검색은 라벨 필터 후 grep 방식이다.
- **LogQL**: Loki의 쿼리 언어. `{namespace="notiflex"} |= "error"`처럼 라벨 필터와 텍스트 필터를 조합한다.
