# 3.4 CI 도구 선택

## 상황

3.3에서 수동으로 `gcloud builds submit`을 실행하여 이미지를 빌드했다. 코드가 바뀔 때마다 수동으로 빌드하는 것은 반복적이고 실수할 여지가 크다. 코드를 push하면 자동으로 이미지를 빌드하는 CI(Continuous Integration) 파이프라인이 필요하다.

## 독자가 물어볼 수 있는 질문

- "CI가 뭐야?"
- "GitHub Actions 말고 다른 CI 도구는?"
- "Jenkins나 GitLab CI랑 비교하면 어때?"
- "Cloud Build를 CI로 쓸 수는 없어?"

## 추천: GitHub Actions

GitHub Actions는 GitHub에 내장된 CI/CD 플랫폼이다. 저장소에 YAML 파일 하나를 추가하면 바로 동작한다.

이 프로젝트에 적합한 이유:
- **GitHub 네이티브**: 코드 저장소와 CI가 같은 플랫폼 → 별도 서버 설치/관리 불필요
- **YAML 선언적**: `.github/workflows/ci.yaml` 한 파일로 파이프라인 정의
- **무료 크레딧**: 퍼블릭 저장소 무제한, 프라이빗도 월 2,000분 무료
- **GCP 인증 간편**: `google-github-actions/auth` 액션으로 서비스 계정 키 한 줄 연동

## 비교

| 도구 | 특징 | 장점 | 단점 | Notiflex 적합도 |
|------|------|------|------|----------------|
| **GitHub Actions** | GitHub 내장 CI/CD | 별도 서버 없음, YAML 선언적, 무료 크레딧 | GitHub 종속, 복잡한 파이프라인은 장황 | ★★★ |
| **Cloud Build** | GCP 네이티브 CI | GCP 서비스 통합 최적, Artifact Registry 직결 | GitHub 트리거 설정 별도, UI가 GitHub보다 불편 | ★★ |
| **GitLab CI** | GitLab 내장 CI/CD | 강력한 파이프라인 문법, 자체 Runner | 코드를 GitLab으로 이전 필요 | ★ |
| **Jenkins** | 자체 호스팅 CI | 플러그인 생태계 풍부, 완전한 커스터마이징 | 서버 운영 필요, 설정 복잡 | ★ |

> **Notiflex 맥락**: GitHub에 코드가 있으므로 GitHub Actions가 가장 자연스럽다. Cloud Build는 GCP 네이티브지만 GitHub 이벤트 트리거 설정이 추가로 필요하다. Jenkins는 별도 서버를 운영해야 해서 학습 환경에서는 과도하다.

## 핵심 개념

- **워크플로우(Workflow)**: `.github/workflows/` 디렉터리의 YAML 파일. push, PR 등의 이벤트에 반응하여 자동 실행된다.
- **Job과 Step**: Workflow는 Job으로 구성되고, Job은 Step으로 구성된다. 각 Step은 쉘 명령이나 재사용 가능한 Action을 실행한다.
- **GitHub Secrets**: 저장소 설정에서 관리하는 암호화된 환경변수. CI에서 GCP 서비스 계정 키(`GCP_SA_KEY`)를 안전하게 사용한다.
- **Artifact Registry**: GCP의 컨테이너 이미지 저장소. CI에서 빌드한 이미지를 푸시하고, GKE가 이 이미지를 풀(pull)한다.
