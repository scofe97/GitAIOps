# 3.3 ArgoCD로 첫 번째 배포 변경 경험

## 상황

3.2에서 ArgoCD를 설치하고 Notiflex가 클러스터에서 동작 중이다. 이제 코드를 수정하고 실제로 배포가 업데이트되는 과정을 경험하려 한다.

## 독자가 물어볼 수 있는 질문

- "코드를 수정하면 어떻게 배포에 반영돼?"
- "Rolling Update가 뭐야?"
- "배포 중에 서비스가 끊기지 않아?"
- "배포 후 롤백은 어떻게 해?"

## 왜 이 방식인가

이 시점에서는 Kubernetes Deployment의 기본 배포 방식인 **Rolling Update**를 사용한다.

- **Deployment 기본값**: 별도 설정 없이 K8s Deployment가 자동으로 Rolling Update를 수행
- **ArgoCD와의 첫 경험**: Git push → ArgoCD 감지 → 자동 배포를 처음 경험하는 단계
- **5장에서 진화**: Rolling Update의 한계(배포 중 일부 에러 노출)를 경험한 후, 5장에서 Blue/Green → 6장에서 Canary로 점진 진화

## 진행 미리보기

```
코드 수정 (v0.1.0 → v0.1.1)
    ↓
이미지 빌드 (gcloud builds submit)
    ↓
매니페스트 이미지 태그 변경 (YAML 수정)
    ↓
git push
    ↓
ArgoCD 자동 감지 → Sync
    ↓
Rolling Update: 새 Pod 생성 → 헬스체크 → 기존 Pod 종료
    ↓
롤백 테스트: ArgoCD에서 이전 커밋으로 Sync
```

## 핵심 개념

- **Rolling Update**: 새 Pod을 하나씩 생성하고 기존 Pod을 하나씩 종료하는 방식. `maxSurge`(초과 생성 수)와 `maxUnavailable`(동시 종료 수)로 속도를 제어한다.
- **ArgoCD Sync**: Git 저장소의 상태와 클러스터의 상태를 일치시키는 동작. 자동 Sync가 켜져 있으면 Git 변경을 감지하여 자동 실행된다.
- **롤백(Rollback)**: 배포에 문제가 있을 때 이전 버전으로 되돌리는 것. ArgoCD에서는 이전 커밋으로 Sync하면 된다.
