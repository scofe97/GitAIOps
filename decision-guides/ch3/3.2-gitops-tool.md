# 3.2 GitOps 도구 선택

## 상황

독자가 2장에서 `kubectl apply`로 수동 배포를 완료했다. 앱은 돌아가지만:
- 누가 언제 어떤 버전을 배포했는지 기록이 없다
- 롤백하려면 이전 매니페스트를 찾아서 다시 apply해야 한다
- 실수로 클러스터를 직접 수정하면 추적이 안 된다

"Git에 push하면 자동으로 배포되고, Git이 유일한 진실이 되는" 방식이 필요하다.

## 독자가 물어볼 수 있는 질문

- "배포를 자동화하고 싶은데 뭘 쓰면 돼?"
- "Git push하면 자동으로 클러스터에 반영되게 할 수 있어?"
- "GitOps가 뭐야?"
- "배포 관리 도구 추천해줘"

## 추천: ArgoCD

ArgoCD는 Git 저장소의 매니페스트를 감시하고, 클러스터 상태를 Git과 자동으로 일치시키는 GitOps 도구다. CNCF Graduated 프로젝트로 커뮤니티가 가장 활발하다.

이 프로젝트에 적합한 이유:
- **Web UI**로 배포 상태를 실시간 확인 → 학습 과정에서 "지금 무슨 일이 일어나는지" 눈으로 볼 수 있다
- **Application CRD**로 "어떤 Git 경로 → 어떤 네임스페이스" 선언적 관리
- **selfHeal**: 누군가 `kubectl edit`으로 직접 수정해도 Git 상태로 되돌린다
- GKE Standard와 네이티브 호환, e2-medium 노드에서 구동 가능 (~500MB 메모리)

## 비교

| 도구 | 특징 | 장점 | 단점 | Notiflex 적합도 |
|------|------|------|------|----------------|
| **ArgoCD** | CRD 기반, Web UI, CNCF Graduated | UI로 배포 상태 시각화, Application 단위 관리, 대시보드 | 메모리 ~500MB, 7개 Pod | ★★★ |
| **Flux** | GitOps Toolkit, CLI 중심, CNCF Graduated | 가볍다(~100MB), HelmRelease CRD, kustomize 네이티브 | UI 없음(별도 Weave GitOps 필요), 학습곡선 | ★★ |
| **Jenkins X** | CI/CD 통합, 파이프라인 포함 | CI와 CD가 하나로 통합 | 매우 무거움, 설치 복잡, 커뮤니티 축소 | ★ |
| **Spinnaker** | 멀티클라우드, Netflix 출신 | 성숙한 생태계, 멀티클라우드 지원 | 설치 복잡(Halyard), 리소스 매우 큼 | ★ |

> **Notiflex 맥락**: e2-medium(4GB) 노드 2개에서 ArgoCD의 500MB는 감당 가능하다. Flux는 가볍지만 UI가 없어서 이 책에서 배포 상태를 눈으로 확인하기 어렵다. 학습 목적에서는 시각적 피드백이 중요하므로 ArgoCD가 최적이다.

## 핵심 개념

- **GitOps**: Git을 Single Source of Truth로 사용하는 운영 방식. 모든 변경은 Git commit으로만 이루어지고, 클러스터는 Git의 상태를 자동으로 반영한다.
- **Reconciliation Loop**: ArgoCD가 3분마다 Git과 클러스터 상태를 비교하고, 차이가 있으면 자동으로 일치시킨다.
- **Self-Heal**: 누군가 `kubectl`로 클러스터를 직접 수정해도, ArgoCD가 Git 상태로 되돌린다. Git이 "유일한 진실"이 되는 기술적 강제.
- **Application CRD**: ArgoCD의 핵심 리소스. "어떤 Git 저장소의 어떤 경로를 어떤 클러스터/네임스페이스에 동기화할 것인가"를 선언한다.
