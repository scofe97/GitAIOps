# 3.5 CI와 CD를 연결하는 구조

## 상황

3.4에서 GitHub Actions로 CI를 만들었고, 3.2에서 ArgoCD로 CD를 구축했다. 하지만 둘이 따로 논다. CI가 이미지를 빌드하면, 수동으로 매니페스트의 이미지 태그를 바꿔야 한다. CI 결과가 자동으로 CD까지 이어지게 하고 싶다.

## 독자가 물어볼 수 있는 질문

- "CI에서 빌드하면 자동으로 배포까지 되게 할 수 있어?"
- "CI-CD 연결 구조를 미리 보여줘"
- "이미지 태그를 Git에서 관리하는 이유가 뭐야?"
- "CI가 직접 kubectl apply 하면 안 돼?"

## 왜 이 방식인가

GitOps에서 CI는 **이미지를 빌드**하고, CD(ArgoCD)는 **Git 상태를 클러스터에 반영**한다. 둘을 연결하는 방법은 "CI가 매니페스트의 이미지 태그를 업데이트하고 git push"하는 것이다.

- **GitOps 원칙 준수**: CI가 `kubectl apply`를 직접 호출하면 Git과 클러스터가 불일치할 수 있다. 매니페스트를 Git으로 업데이트하면 ArgoCD가 감지 → Git이 단일 진실 소스 유지
- **책임 분리**: CI = 빌드/테스트, CD = 배포. 각자의 영역이 명확
- **감사 추적**: 모든 배포 변경이 Git commit으로 남음

## 전체 흐름 미리보기

```
개발자: 코드 수정 → git push
    ↓
GitHub Actions (CI):
  1. 코드 체크아웃
  2. Docker 이미지 빌드
  3. Artifact Registry에 푸시
  4. k8s 매니페스트의 이미지 태그를 새 버전으로 sed 교체
  5. git commit & push (매니페스트 변경)
    ↓
ArgoCD (CD):
  6. 매니페스트 변경 감지
  7. 클러스터에 자동 Sync → 새 Pod 배포
```

> **핵심**: CI가 클러스터를 직접 건드리지 않는다. Git을 통해서만 배포가 이루어진다.

## 핵심 개념

- **이미지 태그 전략**: Git SHA 기반 태그(`abc1234`)를 사용하면 어떤 커밋의 코드가 배포되었는지 추적할 수 있다. `latest` 태그는 어떤 버전인지 알 수 없어 프로덕션에서 쓰지 않는다.
- **CI → Git → CD 패턴**: CI가 매니페스트를 수정하고 git push → ArgoCD가 감지. "Git push triggers"라는 GitOps의 핵심 원칙.
- **GITHUB_ENV**: GitHub Actions에서 Step 간 변수를 전달하는 메커니즘. `echo "VAR=value" >> $GITHUB_ENV`로 설정하면 다음 Step에서 `$VAR`로 접근한다.
