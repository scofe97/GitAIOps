# 7.4 멀티테넌시 격리 방식 선택

## 상황

7.3에서 App of Apps로 여러 앱을 관리하고 있다. 이제 Notiflex를 다른 기업 고객(테넌트)에게도 제공하려 한다. 테넌트 간 데이터와 리소스를 격리해야 한다.

## 독자가 물어볼 수 있는 질문

- "멀티테넌시가 뭐야?"
- "Namespace로 분리하는 것만으로 충분해?"
- "테넌트별로 클러스터를 따로 만들어야 해?"
- "vCluster가 뭐야?"

## 추천: Namespace 분리 + RBAC

테넌트별로 Namespace를 생성하고 RBAC으로 접근을 제한한다. 가장 단순하면서 Kubernetes 기본 기능만으로 구현 가능하다.

이 프로젝트에 적합한 이유:
- **K8s 기본 기능**: 추가 도구 설치 없이 Namespace + RBAC만으로 격리
- **ArgoCD 통합**: 테넌트별 Application을 App of Apps로 관리 → 기존 패턴 재활용
- **리소스 효율**: 단일 클러스터에서 여러 테넌트 운영 → 클러스터 추가 비용 없음
- **학습 가치**: 멀티테넌시의 핵심 개념(격리, 공유 리소스, cross-namespace 통신)을 직접 경험

## 비교

| 방식 | 격리 수준 | 장점 | 단점 | Notiflex 적합도 |
|------|---------|------|------|----------------|
| **Namespace 분리 + RBAC** | 논리적 | 추가 도구 없음, 단순, 리소스 효율적 | 네트워크 격리 약함, 노이지 네이버 가능 | ★★★ |
| **vCluster** | 가상 클러스터 | 테넌트별 가상 Control Plane, 강한 격리 | 추가 리소스, 운영 복잡성 증가 | ★★ |
| **클러스터별 분리** | 물리적 | 가장 강한 격리 | 비용 2배+, 관리 복잡 | ★ |
| **Namespace + NetworkPolicy** | 논리적+네트워크 | 네트워크 수준 격리 추가 | CNI 플러그인 의존, 설정 복잡 | ★★ |

> **Notiflex 맥락**: e2-medium 노드 3개의 단일 클러스터에서 vCluster나 별도 클러스터는 비현실적이다. Namespace 분리로 멀티테넌시의 핵심 개념을 경험하고, cross-namespace DNS(`valkey-primary.notiflex.svc.cluster.local`)로 공유 리소스 접근 패턴을 학습한다.

## 핵심 개념

- **테넌트(Tenant)**: 동일한 인프라를 공유하는 독립적인 사용자 그룹. B2B SaaS에서 각 기업 고객이 하나의 테넌트다.
- **Namespace**: K8s의 논리적 격리 단위. 같은 클러스터 안에서 리소스를 그룹으로 분리한다.
- **RBAC (Role-Based Access Control)**: 누가(ServiceAccount/User) 어떤 리소스에 어떤 동작(get/list/create)을 할 수 있는지 정의한다.
- **cross-namespace DNS**: `<service>.<namespace>.svc.cluster.local` 형식으로 다른 Namespace의 Service에 접근한다. 테넌트가 공유 Valkey에 접근할 때 사용.
- **노이지 네이버(Noisy Neighbor)**: 한 테넌트가 리소스를 과다 사용하여 다른 테넌트에 영향을 주는 문제. ResourceQuota로 완화한다.
