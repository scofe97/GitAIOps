# 7.2 워크로드별 노드 배치 전략 선택

## 상황

모든 워크로드(API, Kafka, Prometheus, CronJob)가 default-pool 2개 노드에 함께 올라가 있다. Kafka처럼 메모리를 많이 쓰는 워크로드가 API Pod의 리소스를 빼앗을 수 있다. 워크로드 특성에 맞게 노드를 분리하고 싶다.

## 독자가 물어볼 수 있는 질문

- "워크로드별로 노드를 분리할 수 있어?"
- "API Pod은 API 전용 노드에만 올리고 싶은데"
- "nodeSelector, taint, affinity 중에 뭐가 좋아?"
- "노드 풀을 추가하려면 어떻게 해?"

## 추천: nodeSelector + 멀티 노드풀

GKE에서 역할별 노드풀(api-pool, worker-pool, ops-pool)을 생성하고, nodeSelector로 Pod을 배치한다. 가장 단순하고 직관적인 방식이다.

이 프로젝트에 적합한 이유:
- **가장 단순**: YAML에 `nodeSelector: {cloud.google.com/gke-nodepool: api-pool}` 한 줄 추가
- **GKE 자동 라벨**: 노드풀 생성 시 `cloud.google.com/gke-nodepool` 라벨이 자동 부여
- **학습 곡선 최소**: taint/toleration보다 이해하기 쉬움

## 비교

| 방식 | 복잡도 | 장점 | 단점 | Notiflex 적합도 |
|------|--------|------|------|----------------|
| **nodeSelector** | 낮음 | 가장 단순, 라벨 매칭만 | "거부" 불가(다른 Pod도 배치됨) | ★★★ |
| **taint/toleration** | 중간 | 노드에 "거부" 걸어 특정 Pod만 허용 | 양쪽 모두 설정 필요 | ★★ |
| **nodeAffinity** | 높음 | required/preferred 유연한 규칙 | 설정 복잡, 학습곡선 | ★ |
| **topology spread** | 높음 | AZ/노드 간 균등 분배 | 단일 존 클러스터에서는 불필요 | ★ |

> **Notiflex 맥락**: 단일 존(asia-northeast3-a) 클러스터에서 3개 노드풀로 분리한다. nodeSelector만으로 충분하다. taint는 "이 노드에 다른 Pod이 올라오는 것을 방지"하고 싶을 때 추가하지만, 학습 환경에서는 과도하다.

## 핵심 개념

- **노드풀(Node Pool)**: GKE에서 동일한 머신 타입과 설정을 가진 노드 그룹. 노드풀별로 머신 타입, Spot 여부, 라벨을 다르게 설정할 수 있다.
- **nodeSelector**: Pod spec에 지정하는 라벨 매칭 조건. 해당 라벨을 가진 노드에만 스케줄링된다.
- **Spot VM**: GCP가 여유 용량을 저렴하게 제공하는 VM. 60~91% 할인이지만, GCP가 언제든 회수할 수 있다. 학습/테스트 환경에 적합.
