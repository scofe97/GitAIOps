# 7.3 여러 앱 한번에 관리하는 패턴 선택

## 상황

7.2에서 노드풀을 분리했고, ArgoCD Application이 이미 여러 개다(notiflex, valkey, monitoring 등). 현재는 각 Application을 개별 YAML로 관리하는데, 새 앱을 추가할 때마다 비슷한 Application YAML을 복사해서 만든다. 여러 Application을 체계적으로 관리하는 패턴이 필요하다.

## 독자가 물어볼 수 있는 질문

- "ArgoCD에서 앱이 여러 개인데 관리가 힘들어"
- "App of Apps 패턴이 뭐야?"
- "ApplicationSet이랑 뭐가 달라?"
- "앱을 한 번에 배포하는 방법 없어?"

## 추천: App of Apps 패턴

하나의 "루트 Application"이 `argocd/` 디렉터리를 감시하고, 그 안에 있는 개별 Application YAML들을 자동으로 배포한다. 디렉터리에 YAML을 추가하면 새 앱이 자동 등록된다.

이 프로젝트에 적합한 이유:
- **가장 직관적**: "폴더에 YAML 넣으면 앱이 생긴다"는 개념이 명확
- **Git 기반**: 루트 앱이 Git 디렉터리를 감시 → GitOps 원칙에 충실
- **기존 구조 활용**: 이미 `argocd/` 디렉터리에 Application YAML이 있음
- **학습 곡선 최소**: 템플릿 문법 없이 순수 YAML만 사용

## 비교

| 패턴 | 복잡도 | 장점 | 단점 | Notiflex 적합도 |
|------|--------|------|------|----------------|
| **App of Apps** | 낮음 | 직관적, 순수 YAML, Git 디렉터리 감시 | 앱마다 개별 YAML 필요 | ★★★ |
| **ApplicationSet** | 중간 | 템플릿으로 대량 생성, 동적 환경 지원 | 템플릿 문법 학습 필요, 디버깅 어려움 | ★★ |
| **수동 관리** | 낮음 | 추가 설정 없음 | 앱 누락 가능, 일괄 관리 불가 | ★ |

> **Notiflex 맥락**: 관리할 앱이 5~7개 수준이므로 App of Apps로 충분하다. ApplicationSet은 "같은 앱을 dev/staging/prod 10개 환경에 배포"하는 경우에 적합하다. Notiflex는 단일 클러스터이므로 템플릿의 이점이 크지 않다.

## 핵심 개념

- **루트 Application**: 다른 Application YAML들이 있는 디렉터리를 source로 지정한 Application. ArgoCD가 이 디렉터리를 감시하여 하위 Application을 자동으로 생성/삭제한다.
- **directory.recurse**: 루트 Application의 source 설정에서 하위 디렉터리까지 재귀적으로 탐색하도록 하는 옵션.
- **selfHeal**: ArgoCD가 클러스터 상태를 Git과 자동으로 일치시키는 설정. 누군가 kubectl로 직접 변경해도 Git 상태로 되돌린다.
- **ApplicationSet**: ArgoCD의 CRD로, 템플릿과 Generator(Git, List, Cluster 등)를 사용하여 여러 Application을 동적으로 생성한다.
