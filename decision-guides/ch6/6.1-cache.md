# 6.1 캐시/상태 공유 도구 선택

## 상황

Notiflex API가 replicas: 2로 동작 중이다. `/id` 엔드포인트는 인메모리 atomic counter로 ID를 생성하는데, Pod마다 독립적인 카운터를 갖고 있어서 같은 ID가 중복 생성된다. Pod 간에 상태를 공유할 중앙 저장소가 필요하다.

## 독자가 물어볼 수 있는 질문

- "Pod이 여러 개인데 데이터를 공유하려면 어떻게 해?"
- "캐시 서버가 필요한데 뭘 쓰면 돼?"
- "Redis 써도 돼?"
- "Valkey가 뭐야?"

## 추천: Valkey

Valkey는 Redis의 오픈소스 포크다. Redis가 2024년 라이선스를 SSPL로 변경하면서 Linux Foundation 산하에서 Valkey가 탄생했다. Redis와 100% 호환되면서 BSD 라이선스를 유지한다.

이 프로젝트에 적합한 이유:
- **BSD 라이선스**: 상용 환경에서도 라이선스 걱정 없이 사용
- **Redis 호환**: 기존 Redis 클라이언트, 명령어를 그대로 사용
- **Bitnami Helm 차트**: `helm install valkey bitnami/valkey`로 간편 설치
- **경량**: standalone 모드로 CPU 50m, Memory 64Mi만 사용

## 비교

| 도구 | 특징 | 장점 | 단점 | Notiflex 적합도 |
|------|------|------|------|----------------|
| **Valkey** | Redis 포크, BSD 라이선스 | Redis 호환, 라이선스 안전, Bitnami 차트 | 커뮤니티 아직 성장 중 | ★★★ |
| **Redis** | 인메모리 DB 표준 | 가장 성숙한 생태계, 풍부한 자료 | SSPL 라이선스(상용 제한) | ★★ |
| **Memcached** | 단순 키-값 캐시 | 매우 가볍다, 멀티스레드 | 영속성 없음, INCR 미지원 | ★ |
| **DragonflyDB** | Redis 호환, 멀티스레드 | 높은 성능, 메모리 효율 | 아직 성숙하지 않음 | ★ |

> **Notiflex 맥락**: ID 생성에 `INCR` 명령이 필요하고, Pod 재시작 후에도 카운터가 유지되어야 한다. Memcached는 영속성이 없어 부적합. Redis는 SSPL 라이선스 문제가 있으므로 Valkey가 최선이다.

## 핵심 개념

- **INCR**: Valkey의 원자적 증가 명령. `INCR notiflex:id`는 키 값을 1 증가시키고 새 값을 반환한다. 여러 Pod이 동시에 호출해도 중복 없이 순차 ID가 생성된다.
- **standalone vs replication**: standalone은 단일 노드로 운영, replication은 master-replica 구조. 학습 환경에서는 standalone으로 충분하다.
- **SSPL (Server Side Public License)**: MongoDB가 만들고 Redis가 채택한 라이선스. 클라우드 서비스로 제공하면 전체 소스를 공개해야 한다. OSI 인정 오픈소스가 아니다.
