# 6.2 Secret 관리 방식 선택

## 상황

6.1에서 Valkey를 설치했고, 비밀번호를 K8s Secret으로 관리하고 있다. 하지만 K8s Secret은 base64 인코딩일 뿐 암호화가 아니며, Git에 커밋할 수 없다. 프로덕션 수준의 Secret 관리가 필요하다.

## 독자가 물어볼 수 있는 질문

- "시크릿을 안전하게 관리하려면 어떻게 해?"
- "K8s Secret을 Git에 넣으면 안 되는 거야?"
- "외부 Secret Manager를 쓰면 뭐가 좋아?"
- "Workload Identity가 뭐야?"

## 추천: Secrets Store CSI Driver + GCP Secret Manager

CSI Driver가 GCP Secret Manager에서 시크릿을 가져와 Pod에 마운트한다. Workload Identity로 SA 키 없이 GKE에서 GCP에 인증한다.

이 프로젝트에 적합한 이유:
- **GKE 네이티브 통합**: Workload Identity가 GKE와 GCP IAM을 직접 연결
- **SA 키 불필요**: JSON 키 파일을 관리할 필요 없음
- **단일 진실 소스**: Secret Manager가 시크릿의 유일한 저장소
- **자동 K8s Secret 생성**: CSI 마운트 시 K8s Secret도 자동 생성

## 비교

| 방식 | 특징 | 장점 | 단점 | Notiflex 적합도 |
|------|------|------|------|----------------|
| **CSI Driver + Secret Manager** | 외부 KMS, CSI 마운트 | GKE 네이티브, SA 키 불필요 | 설정 단계 많음 | ★★★ |
| **Sealed Secrets** | 암호화된 상태로 Git 커밋 | GitOps 호환, 설치 간편 | 키 관리 필요, GCP 통합 없음 | ★★ |
| **External Secrets Operator** | 외부 KMS, K8s Secret 자동 생성 | 다양한 프로바이더, 설정 직관적 | 별도 Operator 설치 | ★★ |
| **kubectl create secret** | K8s 기본 기능 | 가장 간단 | Git에 커밋 불가, base64만 | ★ |

> **Notiflex 맥락**: GKE를 사용하므로 GCP Secret Manager + Workload Identity가 가장 자연스럽다. AWS/Azure를 사용한다면 External Secrets Operator가 더 나을 수 있다. Sealed Secrets는 클라우드 종속성이 없지만, GKE 환경에서는 CSI Driver가 더 깔끔하다.

## 핵심 개념

- **Workload Identity**: GKE Pod의 K8s ServiceAccount를 GCP IAM ServiceAccount와 연결한다. Pod이 GCP 리소스에 접근할 때 SA 키 파일 대신 OIDC 토큰을 사용한다.
- **SecretProviderClass**: CSI Driver의 CRD. "어떤 Secret Manager의 어떤 시크릿을 가져올 것인가"를 정의한다.
- **CSI (Container Storage Interface)**: K8s의 스토리지 플러그인 표준. Secret을 파일 시스템처럼 마운트할 수 있다.
