# 5.3 무중단 배포 전략 선택

## 상황

3장에서 ArgoCD의 Rolling Update로 배포하고 있다. 하지만 Rolling Update는 새 버전에 문제가 있으면 이미 일부 사용자가 에러를 경험한 후에야 알 수 있다. 더 안전한 배포 방식이 필요하다.

## 독자가 물어볼 수 있는 질문

- "배포할 때 서비스 중단 없이 할 수 있어?"
- "Blue/Green 배포가 뭐야?"
- "Rolling Update 말고 더 안전한 방법 없어?"
- "무중단 배포 도구 추천해줘"

## 추천: Argo Rollouts (Blue/Green → 6장에서 Canary로 진화)

Argo Rollouts는 Kubernetes Deployment를 대체하는 CRD로, Blue/Green과 Canary 배포를 선언적으로 구현한다.

이 프로젝트에 적합한 이유:
- **ArgoCD 통합**: 같은 Argo 프로젝트, ArgoCD UI에서 Rollout 상태를 확인 가능
- **CRD 기반**: YAML 선언으로 배포 전략 정의, GitOps 호환
- **점진적 진화**: 5장 Blue/Green → 6장 Canary로 전략을 바꿀 때 Rollout CRD만 수정하면 됨
- **kubectl 플러그인**: `kubectl argo rollouts status`로 배포 진행 상태를 실시간 모니터링

## 비교

| 도구 | 특징 | 장점 | 단점 | Notiflex 적합도 |
|------|------|------|------|----------------|
| **Argo Rollouts** | CRD 기반, ArgoCD 통합 | YAML 선언적, Blue/Green+Canary 모두 지원 | 별도 CRD 설치 필요 | ★★★ |
| **Flagger** | Istio/NGINX/Gateway API 연동 | 메트릭 기반 자동 promote, 다양한 프로바이더 | Istio 없이는 기능 제한, ArgoCD 통합 약함 | ★★ |
| **K8s native Rolling Update** | Deployment 기본 기능 | 추가 설치 없음 | Blue/Green/Canary 불가, 롤백 느림 | ★ |

> **Notiflex 맥락**: ArgoCD를 이미 사용하고 있으므로 같은 Argo 생태계의 Rollouts가 자연스럽다. Flagger는 Istio 없이도 동작하지만, ArgoCD와의 통합이 Argo Rollouts만큼 매끄럽지 않다.

## 핵심 개념

- **Blue/Green**: 새 버전(Green)을 완전히 띄운 후, 트래픽을 한 번에 전환한다. 문제 시 즉시 Blue로 롤백. 리소스 2배 필요.
- **Canary**: 새 버전에 트래픽을 20% → 50% → 80% → 100% 점진적으로 전환. 리소스 효율적.
- **Rollout CRD**: K8s Deployment와 거의 동일하지만, `strategy` 필드에서 `blueGreen` 또는 `canary`를 지정할 수 있다.
- **auto-promote**: Blue/Green에서 설정한 시간(예: 30초) 후 자동으로 Green을 active로 승격한다.
