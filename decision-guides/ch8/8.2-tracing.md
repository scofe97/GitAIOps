# 8.2 분산 트레이싱 도구 선택

## 상황

8.1에서 Kafka를 도입하면서 요청 흐름이 복잡해졌다: API → Kafka → Worker → Valkey. 한 요청이 여러 컴포넌트를 거치는데, 어디서 느린지, 어디서 실패하는지 추적하기 어렵다. 요청의 전체 여정을 시각화할 도구가 필요하다.

## 독자가 물어볼 수 있는 질문

- "요청이 어디서 느린지 어떻게 알아?"
- "분산 트레이싱이 뭐야?"
- "Jaeger 들어봤는데 그걸 써야 해?"
- "Prometheus 로그만으로는 안 돼?"

## 추천: Grafana Tempo

Tempo는 Grafana Labs의 분산 트레이싱 백엔드다. 트레이스를 저장하고 Grafana에서 시각화한다.

이 프로젝트에 적합한 이유:
- **Grafana 통합**: 4장에서 설치한 Grafana에서 바로 트레이스를 조회 → 추가 UI 불필요
- **관측 가능성 3축 통합**: Prometheus(메트릭) + Loki(로그) + Tempo(트레이스)가 Grafana 하나에서 연결
- **경량**: 단일 바이너리 모드, 인덱스 없이 오브젝트 스토리지에 저장
- **OTLP 네이티브**: OpenTelemetry 프로토콜을 기본 지원

## 비교

| 도구 | 특징 | 장점 | 단점 | Notiflex 적합도 |
|------|------|------|------|----------------|
| **Grafana Tempo** | Grafana 네이티브 트레이싱 | Grafana 통합, 경량, OTLP 지원 | Grafana 없이는 UI 없음 | ★★★ |
| **Jaeger** | CNCF 졸업, 자체 UI | 독립 UI, 성숙한 생태계 | 별도 UI 포트, Elasticsearch/Cassandra 필요 | ★★ |
| **Zipkin** | 원조 트레이싱 도구 | 간단한 설치, 자체 UI | 기능 제한, 커뮤니티 축소 | ★ |

> **Notiflex 맥락**: 4장에서 Grafana를 이미 운영하고 있으므로 Tempo가 자연스럽다. Jaeger는 독립적인 UI가 장점이지만, 이미 Grafana가 있는데 또 다른 UI를 추가하는 것은 리소스 낭비다. Tempo + Grafana 조합으로 메트릭 → 로그 → 트레이스를 하나의 대시보드에서 연결하는 경험을 할 수 있다.

## 핵심 개념

- **Trace**: 하나의 요청이 시스템을 통과하는 전체 여정. 여러 Span으로 구성된다.
- **Span**: Trace의 단위 작업. "API 핸들러 처리 50ms", "Kafka 메시지 발행 5ms" 같은 개별 구간.
- **OpenTelemetry (OTel)**: 트레이싱, 메트릭, 로그를 수집하는 벤더 중립 표준. 앱에 OTel SDK를 추가하면 Tempo, Jaeger, Datadog 등 어디로든 보낼 수 있다.
- **OTLP (OpenTelemetry Protocol)**: OTel의 표준 전송 프로토콜. gRPC(4317)와 HTTP(4318) 두 가지 포트를 사용한다.
- **TraceID**: 하나의 요청을 식별하는 고유 ID. 모든 Span이 같은 TraceID를 공유하여 하나의 요청으로 묶인다.
