# 8.3 배치 자동화: 왜 CronJob인가

## 상황

8.2에서 트레이싱까지 구축하여 관측 가능성 3축(메트릭, 로그, 트레이스)이 완성되었다. 이제 운영 자동화가 필요하다. "5분마다 헬스체크를 돌려서 장애를 빨리 발견하고 싶다"는 요구사항이 있다.

## 독자가 물어볼 수 있는 질문

- "주기적으로 실행하는 작업은 어떻게 만들어?"
- "CronJob이 가장 적합한 이유가 뭐야?"
- "외부 스케줄러(Airflow 같은)가 더 낫지 않아?"
- "CronJob이 실패하면 어떻게 돼?"

## 왜 CronJob인가

K8s CronJob은 주기적 작업을 위한 **Kubernetes 기본 리소스**다.

- **추가 설치 없음**: K8s에 내장된 리소스, kubectl로 바로 생성
- **cron 문법**: Linux cron과 동일한 스케줄 표현식 (`*/5 * * * *` = 5분마다)
- **GitOps 호환**: YAML로 선언 → Git에 커밋 → ArgoCD가 관리
- **자동 재시도**: `restartPolicy: OnFailure`로 실패 시 자동 재시작
- **히스토리 관리**: 성공/실패 Job의 보관 개수를 설정하여 리소스 정리

외부 스케줄러(Airflow, Argo Workflows)는 복잡한 DAG(의존성 있는 다단계 워크플로우)에 적합하지만, "5분마다 curl 실행" 같은 단순 주기 작업에는 과도하다.

## 다른 방식과의 차이

| 방식 | 적합한 경우 | Notiflex 헬스체크에 | 이유 |
|------|-----------|-------------------|------|
| **K8s CronJob** | 단순 주기 작업 | 최적 | 설치 불필요, YAML 한 파일 |
| **Argo Workflows** | 복잡한 DAG, 단계별 의존성 | 과도 | 별도 CRD, 단순 작업에 불필요 |
| **Airflow** | 데이터 파이프라인, ETL | 과도 | 별도 서버, 파이썬 생태계 |
| **외부 cron (VM)** | K8s 밖의 스케줄링 | 부적합 | K8s 밖에서 관리, GitOps 불가 |

## 진행 미리보기

```
k8s/smb/healthcheck-cronjob.yaml 생성
    ↓
스케줄: */5 * * * * (5분마다)
    ↓
동작: curl로 Notiflex API /health 호출
  - 200 OK → 성공 로그
  - 그 외 → 실패 (exit 1, restartPolicy: OnFailure)
    ↓
ArgoCD App of Apps로 관리
    ↓
kubectl get cronjob → Job 실행 확인
kubectl logs job/<name> → 결과 확인
```

## 핵심 개념

- **CronJob**: 지정된 스케줄에 따라 Job을 생성하는 K8s 리소스. 매 실행마다 새 Job → Pod이 생성되고, 완료 후 종료된다.
- **Job**: 한 번 실행되고 종료되는 K8s 워크로드. CronJob이 스케줄에 따라 Job을 생성한다.
- **successfulJobsHistoryLimit**: 성공한 Job을 몇 개까지 보관할지. 3이면 최근 3개만 남기고 오래된 것은 자동 삭제.
- **restartPolicy: OnFailure**: 컨테이너가 실패(exit code ≠ 0)하면 같은 Pod에서 재시작. `Never`는 실패 시 새 Pod을 생성한다.
