# 7.2 멀티 노드풀

## 사전 조건
- Workload Identity 활성화 상태 (ch6.2에서 활성화)
  → 모든 새 노드풀에 `--workload-metadata=GKE_METADATA` 옵션 필수
  → 빠뜨리면 해당 노드의 Pod이 GCP Secret Manager 등 GCP API 접근 불가
- vCPU/SSD 쿼터 확인
  → `gcloud compute regions describe <REGION>` 으로 CPUS, SSD_TOTAL_GB 쿼터 사전 확인
  → SSD 쿼터 부족 시 `--disk-type=pd-standard` 사용
- **클러스터 operation 완료 확인 필수**
  → ch6.2에서 Workload Identity, CSI Secret Manager 활성화 등 cluster update operation이 진행 중일 수 있다.
  → operation이 실행 중인 상태에서 노드풀을 생성하면 `"Cluster is running incompatible operation"` 에러가 발생한다.
  → 노드풀 생성 전 반드시 확인:
  ```bash
  gcloud container operations list --zone=<ZONE> --filter="status=RUNNING"
  # 출력이 없어야 정상. 있으면 완료될 때까지 대기한다.
  ```

## 실행 지침

### 단계 1: 역할별 노드 풀 생성

```bash
# API 전용
gcloud container node-pools create api-pool \
  --cluster=notiflex-cluster --zone=<ZONE> \
  --machine-type=e2-medium --disk-type=pd-standard --disk-size=50 \
  --num-nodes=1 --spot \
  --workload-metadata=GKE_METADATA

# 워커 전용 (Kafka 등)
gcloud container node-pools create worker-pool \
  --cluster=notiflex-cluster --zone=<ZONE> \
  --machine-type=e2-standard-2 --disk-type=pd-standard --disk-size=50 \
  --num-nodes=1 --spot \
  --workload-metadata=GKE_METADATA

# 운영 도구 전용
gcloud container node-pools create ops-pool \
  --cluster=notiflex-cluster --zone=<ZONE> \
  --machine-type=e2-small --disk-type=pd-standard --disk-size=50 \
  --num-nodes=1 --spot \
  --workload-metadata=GKE_METADATA
```

> ⚠️ **`--node-labels` 사용 금지**: `--node-labels=role=api` 같은 커스텀 라벨을 추가하지 않는다. GKE는 노드풀 이름을 `cloud.google.com/gke-nodepool: <pool-name>` 라벨로 자동 부여한다. 커스텀 라벨을 추가하면 nodeSelector에서 어떤 키를 써야 하는지 혼란이 생겨, 이후 단계에서 `workload: api`, `role: api` 같은 잘못된 키를 사용하는 원인이 된다.

> ⚠️ **Workload Identity 메타데이터**: ch6.2에서 Workload Identity를 활성화했으므로, 새 노드풀에도 `--workload-metadata=GKE_METADATA`를 설정해야 한다. 이 옵션이 없으면 해당 노드의 Pod가 GCP Secret Manager 등에 접근할 수 없다.

> ⚠️ **release channel 충돌**: GKE 클러스터가 release channel에 등록되어 있으면 `--no-enable-autoupgrade` 옵션을 사용할 수 없다. 이 옵션 없이 노드 풀을 생성한다.

> ⚠️ **Spot VM preemption**: Spot VM은 GCP가 언제든 회수할 수 있다. 노드가 NotReady 상태가 되면 kube-dns, ArgoCD 등에 영향을 줄 수 있다.

> ⚠️ **vCPU 쿼터**: 노드 풀을 여러 개 만들면 쿼터를 초과할 수 있다. 사전에 확인:
> ```bash
> gcloud compute regions describe <REGION> --format="value(quotas[name=CPUS])"
> ```

> ⚠️ **SSD 쿼터 초과**: GCP 기본 SSD 쿼터(`SSD_TOTAL_GB`)는 리전당 300GB이다. 노드 풀이 기본 SSD(pd-ssd) 디스크를 사용하면 쿼터를 초과할 수 있다. `--disk-type=pd-standard --disk-size=50`을 명시하여 HDD를 사용한다.

### 단계 2: nodeSelector 설정

Rollout에 nodeSelector를 추가하여 api-pool에 배치:
```yaml
spec:
  template:
    spec:
      nodeSelector:
        cloud.google.com/gke-nodepool: api-pool
```

> ⚠️ **nodeSelector 키 규칙**: 이 프로젝트의 모든 매니페스트(smb rollout, enterprise rollout, CronJob 등)에서 nodeSelector 키는 반드시 `cloud.google.com/gke-nodepool: <pool-name>` 형식만 사용한다. `workload`, `role`, `tier` 등 임의의 커스텀 키를 사용하면 노드에 해당 라벨이 없어 Pod이 영구 Pending 상태가 된다. 값은 풀 이름 그대로 사용한다: `api-pool`, `worker-pool`, `ops-pool`.

> ⚠️ **nodeSelector 변경 반영**: `kubectl argo rollouts restart`만으로는 nodeSelector가 적용되지 않을 수 있다. 이미지 태그가 동일하면 새 ReplicaSet을 만들지 않기 때문이다. `kubectl apply`로 Rollout spec을 다시 반영하면 Canary 과정을 거쳐 새 ReplicaSet이 생성되면서 nodeSelector가 적용된다.

배치 확인:
```bash
kubectl get pods -n notiflex -o wide
```

NODE 컬럼에서 api-pool 노드에 배치되었는지 확인한다.

## 트러블슈팅

### NotReady 노드
→ Spot VM이 회수된 경우. drain 후 Pod 재배치:
```bash
kubectl drain <node-name> --ignore-daemonsets --delete-emptydir-data
```
