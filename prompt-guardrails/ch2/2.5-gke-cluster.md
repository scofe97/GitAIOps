# 2.5 GKE 클러스터 생성

## 실행 지침

GKE Standard 클러스터를 생성한다.

### 단계 1: 클러스터 생성

현재 gcloud 설정에서 프로젝트 ID와 리전을 확인한 뒤, 아래 조건으로 클러스터를 생성한다:

- **클러스터 이름**: `notiflex-cluster`
- **타입**: GKE Standard (Zonal)
- **존**: gcloud 기본 설정의 zone 사용 (없으면 `asia-northeast3-a`)
- **노드**: e2-medium, 2개, Spot VM
- **Gateway API**: `--gateway-api=standard` 옵션 필수 (5장에서 사용)
- **서브넷**: 기본 VPC 사용

```bash
gcloud container clusters create notiflex-cluster \
  --zone=asia-northeast3-a \
  --machine-type=e2-medium \
  --num-nodes=2 \
  --spot \
  --gateway-api=standard \
  --disk-size=30
```

> ⚠️ `--gateway-api=standard`를 빠뜨리면 5장에서 Gateway API를 사용할 수 없다. 나중에 `gcloud container clusters update --gateway-api=standard`로 활성화할 수 있지만 수 분이 소요된다.

### 단계 2: kubeconfig 컨텍스트 확인

> ⚠️ **로컬 클러스터가 있는 환경**: 로컬에 다른 쿠버네티스 클러스터(minikube, kind, kubeadm 등)가 있으면, GKE 클러스터 생성 후에도 kubectl이 로컬 클러스터를 가리킬 수 있다. 반드시 아래 명령으로 GKE 컨텍스트를 설정한다:

```bash
gcloud container clusters get-credentials notiflex-cluster --zone=asia-northeast3-a
```

컨텍스트 이름을 알아보기 쉽게 변경한다:
```bash
# 이전 실행에서 같은 이름이 남아있으면 먼저 삭제
kubectl config delete-context gke-sysnet4admin_book_gitaiops 2>/dev/null || true
kubectl config rename-context \
  $(kubectl config current-context) \
  gke-sysnet4admin_book_gitaiops
```

> ⚠️ **컨텍스트 이름 충돌**: 이전에 클러스터를 생성했다 삭제한 경우, kubeconfig에 같은 이름의 컨텍스트가 남아있을 수 있다. `rename-context`가 `already exists` 에러를 내면 기존 컨텍스트를 먼저 삭제한다.

### 단계 3: 클러스터 상태 확인

```bash
kubectl get nodes
```

모든 노드가 `Ready` 상태인지 확인한다.

## 트러블슈팅

### 쿼터 초과
`ZONE_RESOURCE_POOL_EXHAUSTED` 또는 vCPU 쿼터 에러가 발생하면:
- 다른 zone으로 변경 (예: `asia-northeast3-b`)
- 또는 GCP 콘솔에서 쿼터 증가 요청

### Gateway API 활성화 안 됨
클러스터 생성 후 GatewayClass가 없으면:
```bash
gcloud container clusters update notiflex-cluster --zone=asia-northeast3-a --gateway-api=standard
```
활성화까지 2-3분 소요.

### 클러스터 삭제 시 GCP 리소스 정리

> ⚠️ **Gateway API 리소스 누수**: GKE Gateway API가 생성한 **forwarding rules**와 **static addresses**는 클러스터를 삭제해도 자동으로 삭제되지 않는다. 이 리소스가 누적되면 `STATIC_ADDRESSES` 및 `IN_USE_ADDRESSES` 쿼터를 초과하여 다음 클러스터에서 Gateway IP 할당이 실패한다.

클러스터 삭제 전에 반드시 정리:
```bash
# 1. Gateway 관련 리소스 먼저 삭제
kubectl delete gateway --all -A
kubectl delete httproute --all -A

# 2. 잠시 대기 (GCP가 로드밸런서를 정리하는 시간)
sleep 30

# 3. 클러스터 삭제
gcloud container clusters delete notiflex-cluster --zone=asia-northeast3-a --quiet

# 4. 누수된 리소스 확인 및 정리 (의존 순서: forwarding-rules → target-http-proxies → url-maps → addresses)
gcloud compute forwarding-rules list --filter="name~gkegw"
gcloud compute addresses list --filter="name~gkegw"
# 남아있는 리소스가 있으면 의존 역순으로 삭제:
# gcloud compute forwarding-rules delete <NAME> --region=<REGION> --quiet
# gcloud compute target-http-proxies delete <NAME> --region=<REGION> --quiet
# gcloud compute url-maps delete <NAME> --region=<REGION> --quiet
# gcloud compute addresses delete <NAME> --region=<REGION> --quiet

# 5. orphaned backend services 정리 (REGIONAL_EXTERNAL_MANAGED_BACKEND_SERVICES 쿼터 누수 방지)
gcloud compute backend-services list --filter="region:asia-northeast3" --format="value(name)" | while read bs; do
  gcloud compute backend-services delete "$bs" --region=asia-northeast3 --quiet 2>/dev/null
done

# 6. orphaned static addresses 확인 (forwarding-rule 없이 address만 남는 경우가 있음)
gcloud compute addresses list --filter="name~gkegw" --format="value(name,region)"
# RESERVED 상태로 남아있으면 삭제:
# gcloud compute addresses delete <NAME> --region=asia-northeast3 --quiet

# 7. orphaned persistent disks 정리 (SSD_TOTAL_GB 쿼터 누수 방지)
gcloud compute disks list --zones=asia-northeast3-a --filter="-users:*" --format="value(name)" | while read disk; do
  gcloud compute disks delete "$disk" --zone=asia-northeast3-a --quiet
done
```

> ⚠️ **target-http-proxies / url-maps 누수**: Gateway API가 생성한 target-http-proxies와 url-maps도 클러스터 삭제 후 남을 수 있다. 이 리소스들이 남아있으면 새 Gateway 생성 시 `urlMaps not found` 에러가 발생하고 IP가 할당되지 않는다. 삭제 시 **의존 역순**(forwarding-rules → target-http-proxies → url-maps → addresses)으로 삭제해야 한다.

> ⚠️ **orphaned backend services**: Gateway API가 생성한 backend services도 클러스터 삭제 후 남는다. `REGIONAL_EXTERNAL_MANAGED_BACKEND_SERVICES` 쿼터(기본 50)를 초과하면 새 Gateway가 프로비저닝되지 않는다.

> ⚠️ **orphaned static addresses**: forwarding-rule, proxy, url-map이 모두 정리된 후에도 static address만 RESERVED 상태로 남을 수 있다. `STATIC_ADDRESSES` 쿼터(기본 8)를 초과하면 새 Gateway IP 할당이 실패한다. cleanup 마지막에 `gcloud compute addresses list --filter="name~gkegw"`로 반드시 확인한다.

> ⚠️ **orphaned persistent disks**: PersistentVolume이 사용하던 GCE 디스크는 클러스터 삭제 후에도 남는다. 반복 실행 시 `SSD_TOTAL_GB` 쿼터를 초과하여 PVC 생성이 실패한다. 클러스터 삭제 후 사용되지 않는 디스크를 반드시 정리한다. 디스크 조회 시 `--filter="zone:..."` 키는 동작하지 않으므로 `--zones=asia-northeast3-a`를 사용한다.
