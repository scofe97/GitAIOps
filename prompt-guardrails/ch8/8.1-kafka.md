# 8.1 이벤트 드리븐: Kafka

## 사전 조건
- worker-pool 존재 (ch7.2에서 생성)
  → Strimzi operator + Kafka broker 모두 worker-pool에 배치
  → `kubectl get nodes -l role=worker`로 확인
- `shared/resource-budget.md` 확인: Strimzi operator 200m + Kafka broker 500m = 700m 추가
  → ch7 이후 노드 추가로 여유 충분하지만, worker-pool 미생성 시 default-pool에 스케줄 불가

## 실행 지침

Strimzi로 Kafka를 설치하고 Notiflex를 Producer/Consumer 구조로 변경한다.

### 단계 1: Strimzi 설치

```bash
kubectl create namespace kafka
helm repo add strimzi https://strimzi.io/charts/
helm install strimzi strimzi/strimzi-kafka-operator -n kafka \
  --set 'tolerations[0].key=role' \
  --set 'tolerations[0].value=worker' \
  --set 'tolerations[0].effect=NoSchedule' \
  --set 'nodeSelector.role=worker'
```

> ⚠️ **Strimzi operator 스케줄링**: 7장에서 멀티 노드풀을 생성하면 default-pool의 CPU 여유가 거의 없다 (monitoring 스택이 차지). Strimzi operator를 worker-pool에 배치하도록 nodeSelector를 설정한다. worker-pool에 taint를 설정한 경우 toleration도 필수이다. taint가 없으면 nodeSelector만으로 스케줄 가능하지만, 워크로드 격리를 위해 toleration을 함께 설정하는 것을 권장한다.

> ⚠️ **zsh에서 `--set` 배열 구문 에러**: zsh 환경에서 `--set 'tolerations[0].key=role'` 실행 시 `no matches found: tolerations[0]...` 에러가 발생한다. zsh가 `[0]`을 glob 패턴으로 해석하기 때문이다. 해결 방법:
> 1. `noglob` 사용: `noglob helm install strimzi strimzi/strimzi-kafka-operator -n kafka --set tolerations[0].key=role ...`
> 2. 또는 `--set` 없이 설치 후 operator가 자연스럽게 worker-pool에 스케줄되도록 nodeSelector만 설정
> 3. 또는 values 파일에 tolerations를 YAML로 작성하여 `-f` 옵션으로 전달

### 단계 2: Kafka 클러스터 생성

`k8s/kafka/kafka-cluster.yaml` 파일을 생성한다:

- **모드**: KRaft (ZooKeeper 없음)
- **브로커**: 단일 노드
- **노드 배치**: worker-pool (`nodeSelector: cloud.google.com/gke-nodepool: worker-pool`)
- **Kafka 버전**: 4.1.0 이상 (Strimzi 0.51+는 4.1.0~4.2.0만 지원. 3.x 또는 4.0.0은 `UnsupportedKafkaVersionException` 발생)
- **API 버전**: `kafka.strimzi.io/v1` 사용 (v1beta2는 deprecated)
- **v1 API 주의**: `kafka.strimzi.io/v1`에서는 `spec.kafka.replicas`와 `spec.kafka.resources`가 제거되었다. replicas와 resources는 KafkaNodePool에서만 설정한다.
- **entity-operator**: topicOperator만 활성화. userOperator는 KafkaUser를 사용하지 않으면 **반드시 제거**한다 (CrashLoopBackOff 방지)
- **entity-operator 메모리**: `limits.memory: 512Mi` 이상

> ⚠️ **Strimzi/Kafka 버전 호환성**: Strimzi 0.51+는 Kafka 4.x만 지원한다. Kafka 3.9.0을 지정하면 `UnsupportedKafkaVersionException`이 발생한다.

> ⚠️ **entity-operator 메모리**: 기본 메모리 설정이 작으면 topic-operator, user-operator가 CrashLoopBackOff에 빠진다. `limits.memory`를 512Mi 이상으로 설정한다. entity-operator가 재시작을 반복해도 Kafka 브로커 자체는 정상 동작하므로 토픽 생성이나 메시지 전송에는 영향이 없다.

Kafka 클러스터가 Ready될 때까지 대기:
```bash
kubectl wait kafka/notiflex-kafka --for=condition=Ready --timeout=300s -n kafka
```

### 단계 3: notifications 토픽 생성

```yaml
apiVersion: kafka.strimzi.io/v1
kind: KafkaTopic
metadata:
  name: notifications
  namespace: kafka
  labels:
    strimzi.io/cluster: notiflex-kafka
spec:
  partitions: 3
  replicas: 1
```

### 단계 4: Notiflex Producer/Consumer 변경

Notiflex 앱을 수정한다:
- **Producer**: `/id` 요청 시 ID 생성 후 Kafka `notifications` 토픽에 메시지 전송
- **Consumer**: 백그라운드 goroutine으로 `notifications` 토픽을 구독하여 로그 출력
- **Kafka 브로커 주소**: 환경변수 `KAFKA_BROKER`로 주입 (`notiflex-kafka-kafka-bootstrap.kafka.svc.cluster.local:9092`)
- **의존성**: `go get github.com/IBM/sarama@latest`로 설치 (v1.46.2+에서 `V4_0_0_0`·`V4_1_0_0` 등 Kafka 4.x Version 상수 제공. `go.mod`에 v1.45 이하가 박히면 `undefined: sarama.V4_0_0_0` 빌드 에러 발생)
- **Version 설정**: 브로커가 Kafka 4.1이면 `cfg.Version = sarama.V4_1_0_0`

Rollout에 `KAFKA_BROKER` 환경변수를 추가한다.

### 단계 5: 빌드 및 배포

이미지 버전을 올리고 빌드, Artifact Registry에 푸시, Rollout 이미지 태그 업데이트 후 배포한다.

### 단계 6: 동작 확인

```bash
# ID 생성 요청
curl http://<GATEWAY_IP>/id

# Consumer 로그에서 Kafka 메시지 수신 확인
kubectl logs -l app=notiflex-api -n notiflex --tail=20 | grep -i kafka
```

## 트러블슈팅

### Kafka 클러스터 NotReady
→ `kubectl describe kafka notiflex-kafka -n kafka`로 원인 확인
→ Kafka 버전이 3.x이면 4.x로 변경

### entity-operator CrashLoopBackOff
→ 메모리 부족이면 `limits.memory: 512Mi`로 증가
→ **user-operator**가 CrashLoopBackOff이면 entityOperator에서 `userOperator` 섹션을 제거한다. KRaft 모드 + Strimzi 0.51에서 user-operator가 broker 연결 시 MetadataUpdate timeout이 발생할 수 있다. KafkaUser를 사용하지 않으면 user-operator가 불필요하다.
→ entity-operator가 불안정해도 Kafka 브로커는 정상 동작

### "server misbehaving" DNS 에러
→ Spot VM 환경에서 kube-dns가 불안정할 수 있음
→ DNS 안정화 후에도 에러가 지속되면 Pod 재시작: `kubectl delete pod -l app=notiflex-api -n notiflex`

### Cloud Build 권한 에러 (bucket 접근 실패)
→ `gcloud config get account`로 현재 활성 계정을 확인한다. 여러 계정이 등록되어 있으면 프로젝트 권한이 있는 계정으로 전환한다:
```bash
gcloud config set account <올바른-계정>
```
→ 계정 토큰이 만료되었으면 `gcloud auth login`으로 재인증한다.
