# 8.2 분산 트레이싱: Tempo

## 사전 조건
- ops-pool 존재 (ch7.2에서 생성)
  → Tempo를 ops-pool에 배치 (default-pool은 CPU 여유 없음)
  → `kubectl get nodes -l role=ops`로 확인
- `shared/resource-budget.md` 확인: Tempo 25m 추가, ops-pool에 여유 충분

## 실행 지침

### 단계 1: Tempo 설치

```bash
helm install tempo grafana/tempo -n monitoring -f helm-values/tempo.yaml \
  --set 'tolerations[0].key=role' \
  --set 'tolerations[0].value=ops' \
  --set 'tolerations[0].effect=NoSchedule' \
  --set 'nodeSelector.role=ops'
```

> ⚠️ **Tempo 스케줄링 실패**: 7장 이후 default-pool에 CPU 여유가 없다. Tempo를 ops-pool에 배치하도록 toleration과 nodeSelector를 설정한다. 만약 Tempo가 Pending 상태에서 helm upgrade로 toleration을 추가해도 Pod에 반영되지 않으면, `kubectl delete statefulset tempo -n monitoring --cascade=orphan && kubectl delete pod tempo-0 -n monitoring` 후 `helm upgrade`를 다시 실행한다.

`helm-values/tempo.yaml`:
- OTLP gRPC receiver 활성화 (port 4317)
- 최소 리소스 설정

### 단계 2: Notiflex에 OpenTelemetry SDK 추가

`app/main.go`에 OTel 트레이싱을 추가한다:
- 의존성: `go.opentelemetry.io/otel`, `otlptracegrpc`
- 환경변수: `OTEL_EXPORTER_OTLP_ENDPOINT` (Tempo gRPC 주소)
- 각 HTTP 핸들러에 span 생성
- TracerProvider 초기화 및 shutdown 처리

> ⚠️ **Go/OTel SDK 버전 호환성**: 최신 OTel Go SDK(v1.42+)는 Go 1.25 이상을 요구한다. 2장에서 `golang:1.25-alpine`으로 시작했다면 변경 불필요. 그렇지 않다면 Dockerfile의 golang 베이스 이미지를 `golang:1.25-alpine`으로 변경한다.

### 단계 3: Rollout에 환경변수 추가

```yaml
env:
- name: OTEL_EXPORTER_OTLP_ENDPOINT
  value: tempo.monitoring.svc.cluster.local:4317
```

### 단계 4: 빌드 및 배포

버전을 올리고 빌드, 배포한다.

### 단계 5: Grafana에서 트레이스 확인

Grafana → Explore → Tempo 데이터소스 → Search로 트레이스를 조회한다.

```bash
curl http://<GATEWAY_IP>/id   # 트레이스 생성
```

## 트러블슈팅

### 빌드 실패 (Go 버전)
→ Dockerfile에서 `golang:1.25-alpine` 사용하는지 확인
→ `go.mod`에서 `go 1.25` 이상인지 확인

### 빌드 실패 (go.sum 미동기화)
→ OTel 의존성을 `go.mod`에 추가한 후, `go mod tidy`를 실행하지 않으면 `go.sum`에 새 패키지의 체크섬이 없어 Cloud Build에서 `no required module provides package` 에러가 발생한다.
→ **반드시 로컬에서 `go mod tidy` 실행 후 `go.sum`도 함께 커밋**한다:
```bash
cd app
go mod tidy
cd ..
git add app/go.mod app/go.sum
git commit -m "feat: add OpenTelemetry tracing with Tempo (v0.7.0)"
git push
```
→ Dockerfile에 `COPY go.sum ./`이 있는지도 확인한다. go.sum이 복사되지 않으면 빌드 단계에서 의존성 검증이 실패한다.

### Grafana가 CrashLoopBackOff — "multiple datasources marked as default"
Tempo 데이터소스 ConfigMap을 추가하고 Grafana가 재시작되면 `Only one datasource per organization can be marked as default` 에러로 Pod이 기동 실패할 수 있다. Prometheus(kube-prom-stack)와 Loki가 둘 다 `isDefault: true`로 프로비저닝되어 있을 때 발생한다.
→ ch4.3 가드레일에 `grafana.datasource.isDefault: false`를 Loki helm-values에 선제적으로 넣도록 업데이트되었지만, 기존 설치분에는 이 플래그가 반영되지 않은 상태일 수 있다. 확인:
```bash
kubectl get cm -n monitoring -l grafana_datasource=1 -o yaml | grep -i isDefault
```
→ default가 2개 이상이면 Loki 쪽을 false로 패치하고 Grafana 재시작:
```bash
kubectl patch cm loki -n monitoring --type=json \
  -p='[{"op":"replace","path":"/data/loki-datasource.yaml","value":"apiVersion: 1\ndatasources:\n- name: Loki\n  type: loki\n  url: http://loki.monitoring.svc.cluster.local:3100\n  isDefault: false\n"}]'
kubectl rollout restart deploy -n monitoring -l app.kubernetes.io/name=grafana
```
ConfigMap 이름은 차트 릴리스명에 따라 다르므로 `kubectl get cm -n monitoring`으로 실제 이름을 먼저 확인한다.

### 트레이스가 Grafana에 안 보임
→ Tempo Pod 로그 확인: `kubectl logs -l app.kubernetes.io/name=tempo -n monitoring`
→ OTEL_EXPORTER_OTLP_ENDPOINT 환경변수 확인
→ Grafana에 Tempo 데이터소스가 추가되어 있는지 확인

### "server misbehaving" 에러
→ Spot VM 환경에서 DNS 불안정. Pod 재시작으로 해결:
```bash
kubectl delete pod -l app=notiflex-api -n notiflex
```

### Tempo 차트 deprecated 경고
`helm install tempo grafana/tempo` 실행 시 "WARNING: This chart is deprecated" 경고가 출력된다.
→ 현재는 정상 동작하므로 무시해도 된다. Grafana는 `grafana/tempo-distributed` 차트로 이전을 권장하지만, 단일 바이너리 모드로 충분한 이 프로젝트에서는 기존 차트를 유지한다.
