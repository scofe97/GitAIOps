# 5.2 Gateway API 도입

## 사전 조건
- ch2.5에서 `--gateway-api=standard` 옵션으로 클러스터 생성 확인
  → 없으면 `gcloud container clusters update --gateway-api=standard`로 활성화 (2-3분)

## 실행 지침

GKE Gateway API를 사용하여 Notiflex에 외부 접근 경로를 만든다.

### 단계 1: Gateway API 활성화 확인

GKE에 Gateway API가 활성화되어 있는지 확인한다:
```bash
kubectl get gatewayclass
```
`gke-l7-regional-external-managed`가 있으면 활성화된 상태다.

> ⚠️ GatewayClass가 없으면 2.5에서 `--gateway-api=standard`를 빠뜨린 것이다. 아래 명령으로 활성화:
> ```bash
> gcloud container clusters update notiflex-cluster --zone=<ZONE> --gateway-api=standard
> ```
> 활성화까지 2-3분 소요. Gateway API CRD만 수동 설치(`kubectl apply -f gateway-api/standard-install.yaml`)해서는 GKE의 GatewayClass가 생성되지 않는다.

### 단계 2: Gateway 생성

`k8s/smb/gateway.yaml` 파일을 생성한다:

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: notiflex-gateway
  namespace: notiflex
spec:
  gatewayClassName: gke-l7-regional-external-managed
  listeners:
  - name: http
    port: 80
    protocol: HTTP
    allowedRoutes:
      namespaces:
        from: Same
```

### 단계 3: HTTPRoute 생성

같은 파일 또는 별도 파일로 HTTPRoute를 생성한다:

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: notiflex-route
  namespace: notiflex
spec:
  parentRefs:
  - name: notiflex-gateway
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /
    backendRefs:
    - name: notiflex-api
      port: 80
```

### 단계 4: HealthCheckPolicy 생성

`k8s/smb/healthcheckpolicy.yaml` 파일을 생성한다:

```yaml
apiVersion: networking.gke.io/v1
kind: HealthCheckPolicy
metadata:
  name: notiflex-healthcheck
  namespace: notiflex
spec:
  default:
    checkIntervalSec: 15
    timeoutSec: 5
    healthyThreshold: 1
    unhealthyThreshold: 2
    config:
      type: HTTP
      httpHealthCheck:
        port: 8080
        requestPath: /health
  targetRef:
    group: ""
    kind: Service
    name: notiflex-api
```

> ⚠️ HealthCheckPolicy가 없으면 GKE는 `/` 경로로 헬스체크를 수행하여 "no healthy upstream" 에러가 발생한다. Notiflex의 헬스체크 엔드포인트는 `/health`이므로 반드시 설정해야 한다.

> ⚠️ HealthCheckPolicy CRD는 GKE Gateway 컨트롤러가 활성화된 후에만 사용 가능하다. `kubectl apply` 시 "no matches for kind HealthCheckPolicy" 에러가 나면 GatewayClass가 준비될 때까지 대기한다.

### 단계 5: proxy-only 서브넷 확인

리전 외부 Gateway(`gke-l7-regional-external-managed`)는 proxy-only 서브넷이 필요하다. GKE가 자동 생성하지만, 없으면:

```bash
gcloud compute networks subnets create proxy-only-subnet \
  --purpose=REGIONAL_MANAGED_PROXY \
  --role=ACTIVE \
  --region=asia-northeast3 \
  --network=default \
  --range=172.16.0.0/23
```

> ⚠️ GKE의 auto-mode 서브넷(`10.128.0.0/9`)과 겹치지 않는 대역을 사용한다.

### 단계 6: 접속 확인

Gateway에 IP가 할당될 때까지 대기(1-3분)한 뒤 테스트:
```bash
GATEWAY_IP=$(kubectl get gateway notiflex-gateway -n notiflex -o jsonpath='{.status.addresses[0].value}')
curl http://$GATEWAY_IP/health
```

## 트러블슈팅

### "no healthy upstream" 에러
→ HealthCheckPolicy가 없거나 잘못 설정됨. `/health` 경로와 `8080` 포트를 확인.

### Gateway IP가 할당되지 않음
→ proxy-only 서브넷이 없을 수 있음. 단계 5 참고.

### "no matches for kind HealthCheckPolicy"
→ GKE Gateway 컨트롤러가 아직 준비되지 않음. `kubectl get gatewayclass`로 확인 후 대기.
