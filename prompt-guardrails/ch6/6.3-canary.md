# 6.3 Canary 배포

## 사전 조건
- preview Service 존재 확인 (ch5.3 Blue/Green에서 생성)
  → `kubectl get svc notiflex-api-preview -n notiflex`
  → 없으면 Canary 전략이 InvalidSpec/Degraded 상태로 빠진다
- ArgoCD auto-sync 상태 인지
  → Blue/Green → Canary 전환 시 반드시 Git push 먼저, 그 다음 Rollout 삭제

## 실행 지침

### 단계 1: Blue/Green → Canary 전략 전환

> ⚠️ **중요**: `kubectl apply`로 전략만 변경하면 Argo Rollouts 컨트롤러가 기존 Blue/Green 상태를 유지한 채 새 Pod을 종료할 수 있다. 반드시 아래 순서를 따른다.

> ⚠️ **ArgoCD auto-sync 충돌**: ArgoCD가 auto-sync 상태이면, `kubectl delete rollout` 후 ArgoCD가 git의 이전(B/G) 버전으로 즉시 복원한다. **반드시 git push를 먼저 한 뒤** Rollout을 삭제해야 ArgoCD가 새 Canary 버전을 적용한다.

`k8s/smb/rollout.yaml`을 Canary 전략으로 변경:

```yaml
strategy:
  canary:
    canaryService: notiflex-api-preview
    stableService: notiflex-api
    steps:
    - setWeight: 20
    - pause: {duration: 30s}
    - setWeight: 50
    - pause: {duration: 30s}
    - setWeight: 80
    - pause: {duration: 30s}
```

전환 순서:
```bash
# 1. Canary 변경사항을 먼저 git push
git add k8s/smb/rollout.yaml
git commit -m "Switch to Canary strategy"
git push origin main

# 2. 기존 Rollout 삭제 (cascade로 관련 ReplicaSet도 함께 삭제)
kubectl delete rollout notiflex-api -n notiflex

# ⚠️ preview Service는 삭제하지 않는다!
# Canary 전략에서 canaryService로 notiflex-api-preview를 사용하므로
# 이 서비스가 없으면 Rollout이 InvalidSpec/Degraded 상태에 빠진다.

# 3. ArgoCD가 git의 새 Canary 버전을 자동 적용 (hard refresh로 즉시 반영)
kubectl annotate application notiflex-smb -n argocd argocd.argoproj.io/refresh=hard --overwrite
```

### 단계 2: Canary 배포 테스트

앱 버전을 올리고 배포한다. Canary 진행 상황을 모니터링:

```bash
kubectl argo rollouts get rollout notiflex-api -n notiflex -w
```

20% → 50% → 80% → 100%로 점진적으로 트래픽이 이동하는 것을 확인한다.

### 단계 3: 커밋

변경 사항을 커밋하고 푸시한다.

## 트러블슈팅

### replicas=1 + Valkey 의존성에서 순환 Pending
→ Canary 전략은 stable(1) + canary(1) = 2 Pod이 필요하다. CPU가 부족하면:
  1. Canary Pod Pending → CPU 확보를 위해 다른 Pod(예: Valkey)도 재시작됨
  2. Valkey Pending → Canary Pod이 떠도 Valkey 연결 실패 → CrashLoopBackOff
  3. 교착 상태: Canary가 Ready되지 않으면 old stable도 내려가지 않음
→ **해결**: `kubectl argo rollouts promote --full`로 즉시 승격 후, old ReplicaSet을 수동 scale=0:
```bash
kubectl argo rollouts promote --full notiflex-api -n notiflex
kubectl scale replicaset <old-rs-name> -n notiflex --replicas=0
```
→ old stable이 내려가면 CPU 확보 → Valkey 복구 → Canary Pod 정상 기동
