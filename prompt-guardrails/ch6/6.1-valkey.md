# 6.1 Valkey 캐시

## 사전 조건
- ArgoCD auto-sync + selfHeal 활성화 상태 (ch3.2)
  → `kubectl apply`로 직접 변경하면 ArgoCD가 Git 상태로 되돌린다. 반드시 Git 경유 배포.
- `shared/resource-budget.md` 확인: ch4 이후 CPU 잔여 ~2415m, Valkey 50m 추가 시 여유 부족 가능
  → Valkey resourcesPreset=none 필수, requests 최소화

## 실행 지침

### 단계 1: Valkey 설치

```bash
helm repo add bitnami https://charts.bitnami.com/bitnami
helm install valkey bitnami/valkey -n notiflex \
  --set architecture=standalone \
  --set primary.resourcesPreset=none \
  --set primary.resources.requests.cpu=50m \
  --set primary.resources.requests.memory=64Mi \
  --set primary.resources.limits.cpu=200m \
  --set primary.resources.limits.memory=128Mi
```

### 단계 2: Notiflex 앱 수정

`app/main.go`를 수정한다:
- 인메모리 카운터 → Valkey(INCR 명령)로 ID 생성
- 의존성: `github.com/valkey-io/valkey-go`
- 환경변수: `VALKEY_ADDR`, `VALKEY_PASSWORD`
- **연결 재시도 로직 필수**:
  ```go
  // 10회 재시도, 3초 간격
  for i := 0; i < 10; i++ {
      client, err = valkey.NewClient(...)
      if err == nil { break }
      log.Printf("Valkey 연결 재시도 %d/10: %v", i+1, err)
      time.Sleep(3 * time.Second)
  }
  ```

> ⚠️ **재시도 로직이 없으면** Pod 시작 시 DNS가 준비되지 않았거나 Valkey가 먼저 기동하지 않은 경우 앱이 즉시 종료되어 CrashLoopBackOff에 빠진다. 실제 테스트에서 3회 `context deadline exceeded` 후 4번째에 정상 연결되는 것을 확인했다.

### 단계 3: Rollout에 환경변수 추가

> ⚠️ **Helm 생성 Secret 이름**: `helm install valkey bitnami/valkey`가 생성하는 Secret 이름은 `valkey`이고, key는 `valkey-password`이다. 가드레일의 예시와 다를 수 있으므로 `kubectl get secret -n notiflex`로 실제 이름을 확인한다.

```yaml
env:
- name: VALKEY_ADDR
  value: valkey-primary.notiflex.svc.cluster.local:6379
- name: VALKEY_PASSWORD
  valueFrom:
    secretKeyRef:
      name: valkey
      key: valkey-password
```

### 단계 4: 빌드 및 배포

> ⚠️ **go.sum 복사**: 의존성을 추가한 후에는 `go.sum` 파일도 생성되므로, Dockerfile의 `COPY go.mod ./` 를 `COPY go.mod go.sum ./`으로 변경해야 한다. (2장에서 `golang:1.25-alpine`으로 시작했다면 Go 버전 불일치는 발생하지 않는다.)

버전을 올리고 빌드, 배포한다. 배포 후 여러 Pod에서 `/id`를 호출하여 ID가 순차적으로 증가하는지(모든 Pod에서 동일한 값을 공유하는지) 확인한다.

### Bitnami 이미지 접근 제한 경고
Bitnami 차트 설치 시 "Since August 28th, 2025, only a limited subset of images/charts are available for free" 경고가 출력될 수 있다.
→ 현재는 경고만 출력되고 설치는 정상 진행된다. 향후 Bitnami 이미지가 유료화되면 Docker Hub 공식 `valkey/valkey` 이미지로 전환을 고려한다.

## 트러블슈팅

### Pod CrashLoopBackOff
→ `kubectl logs <pod-name> -n notiflex`로 확인
→ 재시도 로직이 없으면 추가
→ Valkey 주소/비밀번호가 올바른지 확인

### Valkey Pod Pending (CPU 부족)
→ 2노드 e2-medium 환경에서 4장의 관측 가능성 스택(Prometheus, Grafana, Loki, Fluent Bit) 설치 후 CPU requests 여유가 거의 없다. Bitnami Valkey의 기본 `resourcesPreset`이 높은 리소스를 요청하여 스케줄링에 실패한다.
→ 해결: `--set primary.resourcesPreset=none`으로 기본 preset을 비활성화하고 직접 리소스를 지정한다.
→ 설치 후에도 Pending이면: `kubectl describe pod valkey-primary-0 -n notiflex`로 원인 확인.

### Valkey 연결 실패
→ `kubectl get pods -n notiflex`로 Valkey Pod 상태 확인
→ DNS 확인: `kubectl run test --rm -it --image=busybox -- nslookup valkey-primary.notiflex.svc.cluster.local`

### B/G replicas:2 + Valkey로 CPU 부족
B/G 전략에서 replicas:2이면, 배포 시 active(2) + preview(2) = 4 Pod이 동시에 필요하다. Valkey(50m)까지 추가하면 e2-medium 2노드에서 CPU가 부족해진다.
→ **ch6 진입 전에 replicas를 1로 축소**한다 (Git 경유 필수, kubectl patch는 ArgoCD가 되돌림):
```yaml
# k8s/smb/rollout.yaml
spec:
  replicas: 1
```
→ Git push 후 ArgoCD sync를 확인한다.
→ ch7에서 노드풀을 추가한 후 replicas:2로 복원한다.

### ArgoCD selfHeal 충돌
→ `kubectl apply`로 Rollout을 직접 변경하면 ArgoCD selfHeal이 Git 상태로 되돌린다.
→ **반드시 Git에 커밋/푸시 후 ArgoCD가 동기화하도록 한다.**
→ 즉시 반영이 필요하면 `kubectl annotate application notiflex-smb -n argocd argocd.argoproj.io/refresh=hard --overwrite`
