# 6.2 Secret 관리

## 사전 조건
- Workload Identity 활성화 확인 (ch2.5 또는 이 단계에서 활성화)
  → `gcloud container clusters describe`로 workloadPool 확인
  → 새 노드풀 생성 시에도 `--workload-metadata=GKE_METADATA` 필수
- `shared/resource-budget.md` 확인: GKE managed CSI DaemonSet 추가 시 CPU 영향 있음
  → 2노드 e2-medium에서 ch4 관측 가능성 스택 이후 CPU 90%+ 도달 가능
  → GKE managed CSI는 오픈소스보다 가볍지만 여전히 여유 부족 가능

## 실행 지침

### 단계 0: Workload Identity 활성화

클러스터와 노드풀에 Workload Identity가 활성화되어 있는지 확인한다:
```bash
gcloud container clusters describe <CLUSTER> --zone=<ZONE> --format='value(workloadIdentityConfig.workloadPool)'
```

비어있으면 활성화:
```bash
gcloud container clusters update <CLUSTER> --zone=<ZONE> --workload-pool=<PROJECT_ID>.svc.id.goog
gcloud container node-pools update <POOL> --cluster=<CLUSTER> --zone=<ZONE> --workload-metadata=GKE_METADATA
```

> ⚠️ 클러스터 업데이트는 3~5분, 노드풀 업데이트는 2~3분 소요된다.
> ⚠️ **Workload Identity 활성화 순서와 지연**: 클러스터와 노드풀 **양쪽 모두** 활성화해야 동작한다. 클러스터만 활성화하고 노드풀을 빠뜨리면 CSI Secret 마운트 시 인증 실패가 발생한다. 또한 각 업데이트는 비동기로 진행되므로, 클러스터 업데이트 완료를 확인한 후 노드풀 업데이트를 순차 실행한다. 전체 소요 시간은 **5~10분**이며, 이 동안 노드가 재시작될 수 있다.

### 단계 1: GKE Secret Manager CSI 활성화

GKE managed CSI driver를 사용한다 (오픈소스 helm 설치가 아닌 GKE addon):
```bash
gcloud container clusters update <CLUSTER> --zone=<ZONE> --enable-secret-manager
```

> ⚠️ **GKE managed vs 오픈소스 CSI 이름 차이**:
> - GKE managed: driver=`secrets-store-gke.csi.k8s.io`, provider=`gke`
> - 오픈소스: driver=`secrets-store.csi.k8s.io`, provider=`gcp`
> - SecretProviderClass와 볼륨 설정에서 반드시 GKE 이름을 사용한다.

### 단계 2: Google Secret Manager 설정

Valkey 비밀번호를 Google Secret Manager에 저장하고, Workload Identity로 접근하도록 설정한다.

**Secret 생성 또는 업데이트** (기존 Secret이 있으면 create가 실패하므로 분기 처리):
```bash
# Valkey에 설정한 비밀번호를 변수에 저장
VALKEY_PW="<Valkey 설치 시 사용한 비밀번호>"

# Secret이 이미 있으면 새 버전 추가, 없으면 생성
if gcloud secrets describe valkey-password --project=<PROJECT_ID> 2>/dev/null; then
  echo -n "$VALKEY_PW" | gcloud secrets versions add valkey-password --data-file=- --project=<PROJECT_ID>
else
  echo -n "$VALKEY_PW" | gcloud secrets create valkey-password --data-file=- --project=<PROJECT_ID>
fi
```

> ⚠️ **반드시 `echo -n` 사용**: `echo`(개행 포함) 대신 `echo -n`(개행 없음)을 사용한다. 개행 포함 시 WRONGPASS 에러 발생.

### 단계 2-1: SecretProviderClass 생성

**GKE managed CSI는 provider를 `gke`로 설정**해야 한다 (`gcp` 아님):
```yaml
apiVersion: secrets-store.csi.x-k8s.io/v1
kind: SecretProviderClass
metadata:
  name: notiflex-secrets
  namespace: notiflex
spec:
  provider: gke    # ← GKE managed: "gke" (오픈소스는 "gcp")
  parameters:
    secrets: |
      - resourceName: "projects/<PROJECT_ID>/secrets/valkey-password/versions/latest"
        path: "valkey-password"
```

> `secretObjects` 섹션은 사용하지 않는다. K8s Secret 자동 생성보다 파일 마운트가 안정적.

### 단계 2-2: 파일 기반 Secret 패턴

Rollout 볼륨에서 **GKE CSI driver 이름** 사용:
```yaml
volumes:
  - name: secrets
    csi:
      driver: secrets-store-gke.csi.k8s.io    # ← GKE managed (오픈소스는 secrets-store.csi.k8s.io)
      readOnly: true
      volumeAttributes:
        secretProviderClass: notiflex-secrets
```

앱 환경변수:
```yaml
env:
  - name: VALKEY_PASSWORD_FILE
    value: /mnt/secrets/valkey-password
```

앱 코드에서 파일 읽기:
```go
if pwFile := os.Getenv("VALKEY_PASSWORD_FILE"); pwFile != "" {
    if data, err := os.ReadFile(pwFile); err == nil {
        password = string(data)
    }
}
```

> ⚠️ **GCP SA IAM 전파 지연**: `gcloud iam service-accounts create` 직후에 `gcloud secrets add-iam-policy-binding`을 실행하면 "Service account does not exist" 에러가 발생한다. SA 생성 후 5초 정도 대기한 뒤 IAM 바인딩을 실행한다.

## 트러블슈팅

### CSI addon이 추가하는 시스템 DaemonSet CPU (사전 대응 필수)
→ `--enable-secret-manager` 활성화 시, 노드당 2개의 DaemonSet이 자동 배포된다:
  - `csi-secrets-store-gke`: 70m (3컨테이너: node-driver-registrar 10m + secrets-store 50m + liveness-probe 10m)
  - `csi-secrets-store-provider-gke`: 50m (1컨테이너)
→ **2노드 합계 240m CPU가 순식간에 사라진다.**
→ e2-medium 2노드에서 ch4 관측 가능성 스택 설치 후 CPU가 이미 90%+이므로, CSI 활성화 전에 monitoring CPU를 사전에 최소화해야 한다:
  - Prometheus CR: `spec.resources.requests.cpu: 5m` (기본 100m)
  - Alertmanager CR: `spec.resources.requests.cpu: 5m` (기본 25m)
  - Grafana Deployment: container[2] CPU 5m (기본 50m)
  - kube-prometheus-operator: CPU 5m (기본 25m)
  - fluent-bit DaemonSet: CPU 1m (기본 10m)
  - loki: CPU 1m (기본 10m)
→ **GKE managed DaemonSet은 kubectl patch로 CPU를 줄일 수 없다.** GKE 컨트롤러가 원래 값으로 복원한다. 사용자 워크로드 쪽에서 여유를 만들어야 한다.

### CSI DaemonSet Pending (CPU 부족)
→ 2노드 e2-medium에서 4장 관측 가능성 스택 설치 후 CPU requests가 98~99%에 도달한다.
→ CSI Driver DaemonSet(3컨테이너)과 GCP Provider의 CPU requests를 `5m`으로 축소:
```bash
kubectl patch daemonset csi-secrets-store-secrets-store-csi-driver -n kube-system --type=json \
  -p='[{"op":"replace","path":"/spec/template/spec/containers/1/resources/requests/cpu","value":"5m"}]'
kubectl patch daemonset csi-secrets-store-provider-gcp -n kube-system --type=json \
  -p='[{"op":"replace","path":"/spec/template/spec/containers/0/resources/requests/cpu","value":"5m"}]'
```

### Workload Identity 활성화 지연 — 클러스터+노드풀 양쪽 필수
→ `gcloud container clusters update --workload-pool`만 실행하고 노드풀 업데이트를 빠뜨리면, Pod에서 GCP API 호출 시 `metadata server not available` 또는 `permission denied` 에러가 발생한다.
→ 반드시 클러스터 업데이트 **완료 후** 각 노드풀에 `--workload-metadata=GKE_METADATA`를 적용한다:
```bash
# 1) 클러스터 활성화 (3~5분)
gcloud container clusters update <CLUSTER> --zone=<ZONE> \
  --workload-pool=<PROJECT_ID>.svc.id.goog

# 2) 완료 확인
gcloud container clusters describe <CLUSTER> --zone=<ZONE> \
  --format='value(workloadIdentityConfig.workloadPool)'

# 3) 노드풀 활성화 (풀당 2~3분, 노드 재시작 발생)
gcloud container node-pools update default-pool --cluster=<CLUSTER> --zone=<ZONE> \
  --workload-metadata=GKE_METADATA
```
→ 노드풀이 여러 개면 **모든 풀에 적용**해야 한다. api-pool, worker-pool, ops-pool이 이미 있으면 각각 실행한다.
→ 노드풀 업데이트 중 기존 Pod이 재스케줄링될 수 있으므로, `kubectl get pods -A -w`로 모니터링한다.

### CSI + WI 전체 활성화 후에도 노드 CPU 99% — Loki·FluentBit 임시 제거
→ 단계 1~2 완료 후에도 e2-medium × 2 노드에서 CPU가 99%를 유지하여 Rollout이 Pending에 걸리는 경우가 있다. CSI DaemonSet + Valkey + ArgoCD + Prometheus/Grafana/Alertmanager가 동시에 떠 있는 상태다.
→ **ch7.2에서 역할별 노드풀을 추가하기 전까지만** Loki와 FluentBit를 임시로 내려 여유를 만든다. 메트릭/알림은 유지되므로 실습 진행에는 지장이 없다.
```bash
helm uninstall loki -n monitoring
helm uninstall fluent-bit -n monitoring
kubectl --context gke-sysnet4admin_book_gitaiops top nodes
```
→ ch7.2에서 worker/ops 노드풀 추가 후 원래 설정으로 **반드시 복원**한다. 이때 Loki `values.yaml`의 `grafana.datasource.isDefault: false`를 유지해야 ch8.2 Tempo 설치 시 Grafana datasource default 충돌이 재발하지 않는다.
```bash
helm upgrade --install loki grafana/loki -n monitoring -f helm-values/loki.yaml
helm upgrade --install fluent-bit fluent/fluent-bit -n monitoring -f helm-values/fluent-bit.yaml
```

### Secret Manager 개행 문자 (WRONGPASS)
→ `echo "password" | gcloud secrets versions add ...`는 개행문자(`\n`)를 포함시킨다.
→ **`printf`를 사용한다**: `printf "password" | gcloud secrets versions add ... --data-file=-`
→ 이미 저장했다면 새 버전을 printf로 추가하고, CSI Secret을 삭제 후 Pod 재시작:
```bash
kubectl delete secret <csi-secret-name> -n notiflex
kubectl delete pods -l app=notiflex-api -n notiflex
```
